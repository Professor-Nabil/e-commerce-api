I see the confusion! Let me clarify. In Express, the **Global Error Handler** is that "Safety Net" at the bottom of your `app.ts`. Currently, your `errorHandler` middleware looks for an `err.status`. If you just throw a generic `new Error("Insufficient stock")`, it doesn't have a `.status` property, so it defaults to a **500 (Internal Server Error)**.

We want to tell the client: "Hey, this isn't a server crash, you just sent a bad request (400)."

### The "Clean" Way: Custom Error Class

Instead of messy string checks in the middleware, let's create a reusable error class. This is a very common pattern in Node.js backends.

#### 1. Create `./src/utils/appError.ts`

```ts
export class AppError extends Error {
  public status: number;

  constructor(message: string, status: number) {
    super(message);
    this.status = status;
    // Maintains proper stack trace for where our error was thrown
    Error.captureStackTrace(this, this.constructor);
  }
}
```

#### 2. Update `./src/services/cart.service.ts`

Now, instead of `throw new Error(...)`, use your new class:

```ts
import { AppError } from "../utils/appError.js";

// Inside addToCart or updateQuantity:
if (product.stock < totalRequested) {
  throw new AppError(
    `Insufficient stock. Only ${product.stock} items available.`,
    400,
  );
}
```

---

### The "Quick" Way: Modify `error.middleware.ts`

If you don't want to create a new class yet, you can just make your middleware smarter at identifying "User Errors" vs "Server Crashes".

**Update `./src/middlewares/error.middleware.ts`**:

```ts
import { Request, Response, NextFunction } from "express";

export const errorHandler = (
  err: any,
  _req: Request,
  res: Response,
  _next: NextFunction,
) => {
  // 1. Log the error for you (the developer)
  if (process.env.NODE_ENV !== "test") {
    console.error(err.stack);
  }

  // 2. Determine the status code
  // If it's a stock error or "not found", we force a 400 or 404
  let statusCode = err.status || 500;

  if (err.message.includes("Insufficient stock")) {
    statusCode = 400;
  }

  if (err.message.includes("not found")) {
    statusCode = 404;
  }

  // 3. Send the response
  res.status(statusCode).json({
    error: {
      message: err.message || "Internal Server Error",
    },
  });
};
```

### Which one to choose?

- **The Custom Class** is more "professional" and scales better as the project grows.
- **The Middleware Check** is faster to implement right now to get your tests passing.

Since you are a software developer building a solid MVP, I'd recommend the **Custom Error Class** approach. It makes your intention explicit in the service layer.

**Try updating the service with `AppError` and run your test again—you'll see that `res.statusCode` will finally be 400!** 🚀
