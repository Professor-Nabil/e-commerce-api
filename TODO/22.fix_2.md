You hit the nail on the head. Because you are using a Prisma **Enum**, it is case-sensitive and very strict. If your code was looking for `USER` but the database only accepts `CUSTOMER`, Prisma (and your middleware) will trip up.

Since your schema uses `CUSTOMER`, let's align everything to that.

### 1. The Fix: Registering as Admin in the Test

In your `tests/product.test.ts`, make sure you are passing the role as **`ADMIN`** (all caps to match the Enum).

```typescript
it("should allow an ADMIN to create a product", async () => {
  const adminCredentials = {
    email: "admin@test.com",
    password: "password123",
    role: "ADMIN", // Must match the Enum value exactly
  };

  // 1. Register
  await request(app).post("/api/auth/register").send(adminCredentials);

  // 2. Login
  const loginRes = await request(app)
    .post("/api/auth/login")
    .send(adminCredentials);
  const token = loginRes.body.token;

  // 3. Try to create
  const res = await request(app)
    .post("/api/products")
    .set("Authorization", `Bearer ${token}`)
    .send({
      name: "Mechanical Keyboard",
      description: "A very clicky keyboard", // Added description to match schema
      price: 150.0,
      stock: 10,
    });

  expect(res.statusCode).toEqual(201);
});
```

### 2. Update the Auth Service

Make sure your `registerUser` function is actually passing that role to Prisma. Since it's an Enum, Prisma will validate it automatically.

**File:** `src/services/auth.service.ts`

```typescript
export const registerUser = async (userData: any) => {
  const hashedPassword = await bcrypt.hash(userData.password, 10);

  return await prisma.user.create({
    data: {
      email: userData.email,
      password: hashedPassword,
      role: userData.role || "CUSTOMER", // Match your Enum default
    },
    // ... select ...
  });
};
```
